6 ##Formating
7 With more than one person working on this now and stuff, I figured I should establish a coding standard. Feel free to make suggestions.
8 
9 ```C++
10 if(!(3 == 5)) {
11  int theAnswer = 42; // the answer to life, the universe and everything
12  }
13 ```
14 
15 This sample code demonstrates most of the guidelines:
16 * No spaces around parentheses
17 * Space around binary operators (such as ==, +, -, etc.)
18 * No spaces around unary operators (such as !, *, &, etc.)
19 * Opening braces ({) come 1 space after last part of declaration
20 * Closing braces ({) go on their own line at the same indentation level as the block they are closing
21 * all names of objects, classes, functions and variables should be in camel-case (first letter of every word capitalized), no underscores or dashes
22  * for variables and functions, the first letter is lowercase
23  * for objects and classes, the first letter is capitalized
24 * Macros (`#define`) and constants (`const`) are declared in all caps.
25 * comments come 1 space after the relevant line
26 * Use the tab character, not spaces
27 
28 ##Naming Conventions
29 For the most part, simply name stuff what makes the most sense. A few restrictions:
30 * include guards must be named in all caps `FILE_NAME_H`, with underscores to seperate words
31 * arguments being passed to methods simply to be used to set a corisponding variable in the class should be named `tempVariableName`, where `variableName` is the name of the variable in the class that will be set.
32 * functions used to change data in a class should be called `setSomething`, where `Something` describes what is to be set
33 * functions used to access data in a class should be called `getSomething`, where `Something` describes what is to be returned
34 
35 ##Comments
36 The majority of comments should be of the `//` variety, as specified in the formatting section. place them after any confusing line, or to clarify an unclear name.
37 
38 Document all functions and classes! No need to document variables. We are using a documentation program called Doxygen, which generates documentation using comments as specified in doxygenConfig and as used in deploy.sh. Doxygen requires a specific style of comments, as follows:
39 
40 ```C++
41 /**
42  * this is a description about what this class/function does. this description supports Markdown
43  *
44  * @param parameterName this is a description about what a parameter (argument) is for
45  * @param another describe all parameters
46  *
47  * @return what this function returns (exclude this line if it returns nothing)
48  */
49  ```
50 
51 All documentation should be placed in the .h file whenever possible.

53 The documentation can be viewed by opening a browser to file:///path/to/ben/docs/index.html

56 ##Coding Style
57 Each class should be divided into 2 files, a .h and .cpp, with a proper [include guard](http://faculty.cs.niu.edu/~mcmahon/CS241/c241man/node90.html) in the .h file. File names should follow the format classNameWithFirstCharacterLowercase.extension, and include guard macro names should be `CLASS_NAME_WITH_UNDERSCORES_IN_ALL_CAPS_H` (always include the `_H`). The one exception to this rule are template classes, in which case put what would normally be in .cpp below the contents of the .h, with the include guard around the whole file.
58 
59 Do not make unnecessary classes, but if it is likely that a more generalized version of the class will be useful, it is easier to make them both then later do the separation between generalization and specification.
60 
61 In most cases, memory is more expensive then time, so it is encouraged to recalculate some numbers in order to save space.

63 ##Git and Github
64 We are using Git and Github to manage this project. Git is a command line versioning tool to keep track of changes, and Github is a hosted server to store these files on. For the most part, the deploy script should take care of the messy details, but I suggest going through [this](https://try.github.io/levels/1/challenges/1) introduction to git.
65 To learn more about git read [this](http://git-scm.com/book). The only thing it is mandatory to to know to use this project is how to resolve conflicts. So after going through the basic [tutorial](https://try.github.io/levels/1/challenges/1), read at least [this](http://git-scm.com/book/en/Git-Branching-Basic-Branching-and-Merging) page.

67 ##Deploying
68 The deployment script (deploy.sh) takes two options:
69 the first is a flag, indicating which Raspberry Pi the script should attempt to download code onto. The current options are
70 * `-g` for Gabriel's pi
71 * `-k` for Keane's
72 * `-n` for no Pi
73 
74 The second option is the commit message for git. It should be enclosed in quotes.
75 
76 The script does the following (in order)
77 * generates updated documentation
78 * commits changes to Github
79 * if specified, ssh's into a Raspberry Pi, and commands it to pull down new changes from Github using the pullChanges.sh script, then compiles the new code using the Makefile (see the next section for more on that)

81 ##Linking and Compiling Code
82 I have chosen to use Makefiles to compile code. A Makefile describes how an executable depends on different files, and how each of those files in turn how made. One executable file is made from a number of object files being combined. Each object file in turn is made of one .cpp file and any other files it includes (the accompanying .h file at a minimum). You will need to modify the Makefile whenever you add or remove files. You should be able to do that simply by copying and pasting what is already there. To learn more about how they work read through the [manual](https://www.gnu.org/software/make/manual/html_node/index.html).

84 ##What You Need to Contribute to this Project
85 Besides the obvious (computer, intelligence), you need:
86 * the linux operating system
87 * git (`sudo apt-get install git`)
88 * make (`sudo apt-get install make`)
89 * g++ (`sudo apt-get install g++`)
90 * a decent text editor (I suggest [Sublime Text](http://www.sublimetext.com/))
91  * if you get Sublime, some useful packages are:
92  * [Package Control](https://sublime.wbond.net/) is absolutely necessary. Installing it will allow you to install all other packages from inside Sublime.
93  * Diff for git comparisons
94  * DoxyDoc for documenting C++
95  * Git for running git command from inside sublime
96  * GitGutter for viewing changes this commit
97  * My favorite theme is Minimal, but that's a personal preference
98 
99 a Raspberry Pi is needed to test code, because that's the only place where the C++ hardware library will compile. Your pi will need:
100 * an Internet connection
101 * git (`sudo apt-get install git`)
102 * make (`sudo apt-get install make`)
103 * g++ (`sudo apt-get install g++`)
104 * [wiringPi](http://wiringpi.com/)
105 for convenience, give it a static IP address then add a flag to the deployment script.
